## Problem Statement

Microsoft removed OneDrive sync support for on-premise SharePoint. Organizations that cannot
or will not migrate to Microsoft 365 — due to data sovereignty, compliance, air-gapped
networks, or regulated environments — have no sync client for their document libraries.

Users fall back to manually downloading/uploading files through the browser. That is the
status quo DataDrive replaces.

**The reframe:** DataDrive is not a "OneDrive replacement." It is the sync layer for
organizations that chose sovereignty over convenience. "OneDrive for on-prem" explains it to
a user. "The sync layer for data-sovereign organizations" explains it to the CTO who approves
the deployment.

---

## What Makes This Cool

- Files live in Windows Explorer. No browser. Save in Word → uploads to SharePoint automatically.
- Permission-aware tree: you see only what you can read. Folders you cannot write are locked; you
  cannot edit them even in Explorer — enforcement is in the sync engine, not just the UI.
- Works offline. Conflicts produce "keep both" copies — no data loss, ever.
- For domain users via Windows SSO (Kerberos): zero credentials stored, ever. No password prompt.
- Open source (MIT) — any org can audit, self-host, and contribute.

---

## Constraints

- Platform: Windows desktop (Electron + Node/TypeScript).
- SharePoint: on-premise 2016, 2019, Subscription Edition.
- Auth: Windows SSO (NTLM/Kerberos, SSPI) first. Credential Manager fallback for non-SSO paths.
  **No webhooks** on on-prem SP — all remote change detection is polling-based.
- Stack: Electron, PnPjs v4 (`@pnp/sp` + `@pnp/nodejs`), node-sp-auth, chokidar, better-sqlite3,
  keytar (for Credential Manager fallback only).
- Distribution: MIT license. GitHub Releases (signed installer). electron-builder packaging.
- v1 scope: individual end users self-installing. MSI/GPO/IT-admin deployment is Phase 4.
- Files-On-Demand (cfapi) is a Phase 0 spike, not a Phase 1 commitment.
- Non-domain machines (workgroup, contractor laptops on VPN): v1 supports explicit NTLM
  credentials stored in Windows Credential Manager (keytar/DPAPI). Not deferred.

## Hierarchy and Permission Rule

```
Site Collection (root web)
└── Sub-site (Web)              — recursive, any depth
    └── Document Library (List, BaseTemplate 101)
        └── Folder
            └── Document
```

`EffectiveBasePermissions` check per node (evaluated server-side, not inferred from parent):

| Permission state              | Behavior                                        |
|-------------------------------|-------------------------------------------------|
| ViewListItems + Edit/Add      | READ-WRITE mount — two-way sync                 |
| ViewListItems only            | READ-ONLY mount — sync down, block uploads      |
| No ViewListItems              | **Hidden** — not shown, not synced              |

Hidden nodes are invisible to the user. The visible tree is the accessible tree.
Visible-but-locked is deferred to a later phase if requested by users.

**Read-only enforcement:** The local file remains writable on disk (no NTFS ACL changes —
those would require admin rights). Enforcement is at the upload layer: if `permission_level =
'ro'` for a dirty file, the upload is refused, and the user gets a tray notification:
"[filename] was not uploaded — you don't have write access." The local edit is preserved
locally but never sent to SP. A future phase may add NTFS read-only attributes as an optional
setting for users who want stronger local enforcement.

## State Store (SQLite)

One `better-sqlite3` database at `%APPDATA%\DataDrive\state.db`. Schema:

```sql
CREATE TABLE sync_items (
  id               INTEGER PRIMARY KEY,
  server_url       TEXT NOT NULL UNIQUE,   -- SP serverRelativeUrl
  local_path       TEXT NOT NULL,          -- absolute local path
  sp_item_id       INTEGER,               -- SP list item numeric ID; nullable; see Delete Behavior for null fallback
  etag             TEXT,                   -- SP ETag at last sync
  sp_version       INTEGER,               -- SP list item version number
  last_synced_at   INTEGER,               -- Unix epoch ms
  local_mtime      INTEGER,               -- local file mtime at last sync (epoch ms)
  local_size       INTEGER,               -- local file size at last sync
  dirty            INTEGER DEFAULT 0,     -- 1 = local change not yet uploaded
  permission_level TEXT DEFAULT 'rw'      -- 'rw' | 'ro' | 'hidden'
);

CREATE TABLE libraries (
  id            INTEGER PRIMARY KEY,
  site_url      TEXT NOT NULL,
  list_id       TEXT NOT NULL UNIQUE,
  title         TEXT NOT NULL,
  local_root    TEXT NOT NULL,            -- local folder this library maps to
  change_token  TEXT,                     -- current GetChanges token
  last_polled   INTEGER                   -- Unix epoch ms of last poll
);
```

Recovery on DB loss or corruption: rebuild from SharePoint on next launch. A missing DB
triggers a full re-enumerate and re-download (Phase 1 path). Dirty-flag loss means local
edits made since last sync are treated as untracked local files — surface them via conflict
resolution (keep both), do not discard.

## Remote Change Detection (Polling)

On-prem SharePoint has no reliable webhooks. Change detection uses `GetChanges` per library
on an interval.

**Polling loop (per library, runs in main process):**
```
1. Load stored change_token from libraries table.
2. Call: POST /_api/web/lists('<list-id>')/GetChanges
   Body: { query: { ChangeTokenStart: <token>, Add: true, Update: true, Delete: true,
                    Item: true, File: true, Folder: true } }
3. For each changed item in response:
   - Resolve serverRelativeUrl → local_path via sync_items table.
   - Apply download/upload/delete logic (see pipeline below).
4. Persist new ChangeToken from response to libraries.change_token.
5. Sleep <interval>. Repeat.
```

**First poll (no stored token):** pass `ChangeTokenStart: null` — SP returns all current
items. This is the initial populate-from-remote step. During initial sync for a library, no
conflict is possible (no prior `sync_items` row exists) — download wins unconditionally.

**Initial sync concurrency:** when the user adds multiple libraries, initial syncs run
per-library independently and in parallel, capped at 2 concurrent downloads. Libraries
beyond the cap queue.

**Interval target:** 30 seconds per library by default. Configurable in settings (10s–5 min).
See Open Questions for tuning guidance.

**Sync latency target:** a file saved in SharePoint appears locally within 60 seconds under
normal network conditions.

## Conflict Detection and Resolution

**Detection — how we know a conflict exists:**

A conflict occurs when ALL of the following are true for the same file:
1. Remote: SP ETag or version differs from `sync_items.etag` / `sync_items.sp_version`.
2. Local: `local_mtime` or `local_size` differs from the values stored at last sync.

If only (1): remote changed, local unchanged → download (no conflict).  
If only (2): local changed, remote unchanged → upload (no conflict).  
If both (1) and (2): **conflict**.

**Resolution — "keep both" (default, never configurable away from this in v1):**
```
1. Download the remote version → save as original filename (overwrite local).
2. Rename the local version →
     "<basename> (conflict — <username> — <YYYY-MM-DD>).<ext>"
   where <ext> = path.extname(filename) (last extension only)
   and <basename> = filename minus that extension.
   Example: "report.final.docx" → "report.final (conflict — jsmith — 2026-06-30).docx"
3. Upload the renamed local version to SharePoint.
4. Record both in sync_items with their respective etags and sp_item_ids.
5. Notify user via tray: "Conflict in <filename> — both copies saved."
```

The user resolves manually. DataDrive never silently loses data.

## Delete Behavior

Local file deleted → must call SharePoint `recycle()` endpoint, NOT the HTTP DELETE verb.  
HTTP DELETE hard-deletes the file. `recycle()` moves it to the SP Recycle Bin.

```
POST /_api/web/lists('<list-id>')/items(<sp_item_id>)/recycle()
```

Use the stored `sp_item_id` from `sync_items`. If unavailable (e.g., row missing after a DB
rebuild), resolve it first:
```
GET /_api/web/GetFileByServerRelativeUrl('<server_url>')/ListItemAllFields/Id
```
Then call `recycle()` with the returned ID. If the remote file is already gone (404 on the
resolve), log it as "already deleted remotely" and remove the local `sync_items` row.

This is non-negotiable for the success criterion "delete is recoverable."

## Error Taxonomy

Three tiers. Every error is classified at the point it occurs.

| Tier | Description | User signal | Example |
|------|-------------|-------------|---------|
| **Transient** | Network blip, SP momentarily unavailable | Retry silently (exp. backoff) | TCP timeout, 503 |
| **Actionable** | User must do something | Tray notification + activity log entry | 401 re-auth, disk full, conflict |
| **Fatal** | Sync cannot continue | Tray error badge + modal on next open | DB corruption, SP site deleted, auth token permanently revoked |

Transient errors retry with exponential backoff (1s, 2s, 4s, 8s, max 5 min). After 3
consecutive fatal signals, sync pauses and the tray icon shows a red badge.

No error is swallowed silently. Every error writes to the activity log.

---

## Activity Log

- **Location:** `%APPDATA%\DataDrive\logs\datadrive-YYYY-MM-DD.jsonl` (one file per day)
- **Format:** newline-delimited JSON: `{ "ts": "<ISO8601>", "level": "info|warn|error", "event": "<slug>", "file": "<serverRelativeUrl>", "detail": "<message>" }`
- **Retention:** last 7 days or 50 MB total, whichever comes first. Older files are deleted on startup.
- **In-app:** the sync status UI shows the last 100 log entries, filterable by level.

## Auth Flows

**Domain user (SSO — preferred path):**
```
1. App starts → try SSPI/Negotiate against SP site URL.
2. SP returns 200 → authenticated. Store NOTHING. Proceed.
3. SP token expires during session → silent renew via SSPI. If renew fails → Actionable error.
```

**Non-domain user or SSO unavailable (Credential Manager path):**
```
1. SSO fails → prompt: domain\username + password.
2. Validate against /_api/web (HTTP 200 = valid).
3. Store credentials in Windows Credential Manager via keytar (DPAPI-encrypted).
   Never write to disk, localStorage, app config, or SQLite.
4. On AD password change: SP returns 401 → re-prompt, update Credential Manager entry.
```

For domain users, no password is ever stored. For non-domain/SSO-fallback users, credentials
are stored only in Windows Credential Manager (DPAPI), never in plaintext anywhere.

## Local Watcher (chokidar)

One chokidar watcher per synced library, watching `libraries.local_root` recursively.

- **Watched paths:** `libraries.local_root` for each active library. Watching starts on app
  launch for each library and stops when a library is removed.
- **Symbolic links:** not followed. Symlinks inside a local_root are ignored.
- **Exclusion patterns (never uploaded, never trigger sync):**
  - `~$*` — Office lock files (Word, Excel)
  - `*.tmp`, `*.TMP`
  - `desktop.ini`, `Thumbs.db`, `.DS_Store`
  - `*.lnk` — Windows shortcuts
  - Any file or folder whose name starts with `.` (hidden system files)
- **Debounce:** watcher events are debounced 500ms per path before entering the upload queue.
  Rapid saves (e.g., autosave in Word) produce one upload, not many.
- **New local subfolder (`addDir` event):** create the corresponding folder in SharePoint via
  `_api/web/folders` before uploading any files inside it. Do not upload files to a path
  whose parent folder does not yet exist on SP.
- **local_root deleted:** if the user deletes the watched root folder itself, log a Fatal error,
  badge the tray, and pause sync for that library. Do not attempt to recreate the folder.

## Auth + PnPjs Compatibility Note

PnPjs v4 (`@pnp/sp`) is browser-first. Node usage requires `@pnp/nodejs` which provides a
Node fetch adapter. `node-sp-auth` injects NTLM headers by wrapping `node-fetch`. The
critical compatibility risk: does PnPjs v4's `@pnp/nodejs` adapter accept `node-sp-auth`'s
fetch wrapper without conflict?

This is a **required sub-item of Spike A** — not assumed safe. If the two fetch layers are
incompatible, the fallback is: bypass PnPjs for auth and make raw `node-fetch` calls with
`node-sp-auth` headers for NTLM, then pass authenticated cookies/tokens to a PnPjs
`SPBrowser` context. The spike must confirm which path works.

## Approaches Considered

### Approach A: Roadmap As-Is (CHOSEN)
Phase 0 spikes → Phase 1 (read path) → Phase 2 (two-way sync) → Phase 3 (OneDrive feel) → Phase 4 (enterprise)
- Effort: L (Phase 0+1+2 = ~10–12 weeks)
- Risk: Med (cfapi spike is the Phase 3 gate; auth is the Phase 1 gate)
- Pros: Risks surface early via spikes; architecture is cleanest from the start
- Cons: Phase 1 alone requires a passing auth spike first

### Approach B: Community-First Fast Ship
Phase 1 (read-only full-copy) shipped to GitHub immediately without cfapi spike.
- Effort: S–M (2–3 weeks to v0.1 read-only)
- Risk: Low (no cfapi risk)
- Pros: Real users, real SP version diversity, GitHub stars early
- Cons: Upload not in v0.1; momentum risk after Phase 1

### Approach C: Windows Service + Web UI
Node Windows Service for sync; localhost React for config UI; minimal Electron tray.
- Effort: L–XL
- Risk: Med-High (three separate components)
- Pros: Lightest footprint; IT-controllable service
- Cons: Three components to install; distribution complexity

## Open Questions

1. **[HIGH] cfapi spike feasibility.** Can `node-addon-api` call `CfRegisterSyncRoot` +
   `CfCreatePlaceholders` successfully from inside an Electron main process? Spike in Week 1.
   Fallback: full-copy selective sync (still shippable); Files-On-Demand becomes v2.

2. **[MED] Change token polling — full-copy fallback.** If cfapi fails, full-copy Phase 1
   sync has its own constraints: initial sync of a 100k-item library could take hours and
   consume significant disk space. Mitigate by: (a) showing estimated size before sync starts,
   (b) showing progress during initial download, (c) requiring user to confirm libraries > 5 GB.

3. **[MED] node-sp-auth SSO path.** Does it consume a current Kerberos ticket silently?
   Confirm in Spike A. If not, find a Negotiate/SSPI module for zero-credential-stored path.

4. **[MED] Windows path limits.** Long paths (> 260 chars), SP-illegal characters
   (`` ` " # % & * : < > ? / \ { | } ~ ``), URL lengths (~400 chars). All must fail with a
   clear user-facing message and an activity log entry. Never silent.

5. **[MED] Polling interval tuning.** 30s default is unvalidated. Target: measure SP server
   load impact of polling 10 libraries at 30s on a 100k-item farm before shipping. Expose
   the interval in settings (range: 10s–5 min).

6. **[DISTRIBUTION] GitHub release pipeline.** Set up electron-builder + GitHub Actions to
   publish a signed `.exe` installer on git tag push. Without this, there is no install path
   for users who find DataDrive on GitHub. Ship this before Phase 1 goes public.
   Include electron-updater (E9) in this initial pipeline — it's cheap here, expensive later.

## Success Criteria (v1 / Phase 2)

1. Domain user installs, enters site URL, authenticates via Windows SSO — **no password stored**.
2. Picks two libraries: one editable, one read-only.
3. Both appear under a local `DataDrive\` folder.
4. Read-only library: local edits are **blocked** with an error message; changes are **not uploaded**.
5. Edit in Word → save → **a new major version appears in SP's version history** (verifiable via SP web UI; previous version still accessible).
6. Conflict (same file edited locally + remotely since last sync) → a `(conflict — username — date)` copy is produced locally and uploaded; **no data is lost**.
7. Local delete → **file appears in the SP Recycle Bin** (not hard-deleted; verified via SP web UI).
8. Network drop mid-sync → **resumes cleanly on reconnect**; no item is lost or duplicated.
9. Tray icon reflects sync state (syncing / idle / paused / error). Activity log shows last 100 events.
10. **Sync latency:** a file saved to SP appears in the local folder within 60 seconds under normal network conditions.

   sub-item catches that.

   Fail: document the exact failure point and identify the fix (e.g., bypass PnPjs fetch
